# 载体对比与推荐（Browser Extension vs Script vs RPA）

## 1. 载体选型的评价维度

- **合规与可持续**：是否更容易遵循平台规则，是否易被封/风控
- **实现成本**：研发、维护、部署与跨平台成本
- **稳定性**：页面改版适配、异常恢复能力
- **可观测与可审计**：是否容易留痕、回放、解释
- **扩展性**：多账号、多主题、多策略、多团队协作
- **安全性**：凭证存储、权限控制、数据泄露风险

## 2. 主要载体类型对比

### A. 浏览器插件（Chrome/Edge Extension）

**定位**：在真实浏览器环境中辅助浏览与交互；天然适合“人机协同”。

- **优点**
  - 贴近真实交互，天然支持“人工确认”
  - 可直接获取当前页面 DOM 信息用于候选提取与留痕
  - 适合做“评论工作台”“候选池标注”等交互型工具
- **缺点**
  - 受页面结构变化影响较大（需要持续适配）
  - 复杂后台调度/多任务并发不如服务端自然
  - 需要处理权限与敏感数据（cookies/token）的安全存储
- **适用**
  - MVP 首选执行层：浏览/搜索/候选收集/点赞收藏（低风险）
  - 评论强制人工确认的工作流

### B. 后台脚本/定时任务（Server/Local Daemon）

**定位**：策略、调度、队列、报表、审计的“控制平面”。

- **优点**
  - 调度与任务编排能力强（cron、队列、重试、限流）
  - 易做报表、告警、审计、权限与多租户
  - 可独立于 UI 运行，可靠性更好
- **缺点**
  - 若直接“模拟交互”风险高且更容易触发风控（不推荐）
  - 若无官方 API，采集与执行往往仍需依赖浏览器/RPA
- **适用**
  - 作为中枢：存储配置、生成执行计划、下发任务、收集回执

### C. 桌面 RPA（Playwright/Selenium + 桌面封装，或商业 RPA）

**定位**：用自动化驱动浏览器或桌面应用完成流程；可做无人值守但风险更高。

- **优点**
  - 能覆盖“没有 API、页面复杂”的流程自动化
  - 易与截图/录像结合，审计证据强
  - 可部署在专用执行机，隔离凭证
- **缺点**
  - 受页面改版影响最大；稳定性依赖大量工程化治理
  - 无人值守自动互动容易触发平台风控（需极度保守配额/人工兜底）
- **适用**
  - 作为“执行器（Worker）”对接控制平面；优先用于低风险动作

### D. 移动端自动化（Android Accessibility / iOS 自动化）

**定位**：更贴近真实使用环境，但研发成本与合规风险更高。

- **优点**
  - 与真实 App 行为更一致（但不代表更合规）
  - 可覆盖 App 内独占能力（如部分入口只在 App）
- **缺点**
  - iOS 自动化限制多、成本高；系统更新与适配复杂
  - 安全与隐私边界更敏感
  - 运维复杂（真机农场/设备管理）
- **适用**
  - 仅在确有必要且有强合规评审时考虑；不建议作为 MVP

### E. 官方 API / 第三方合规数据服务（如果可用）

**定位**：最合规、最可持续的方式（但能力可能不足）。

- **优点**
  - 稳定、可控、合规风险最低
  - 更适合规模化与企业化
- **缺点**
  - 权限、额度、可获取字段受限；互动能力通常受限或不可用
- **适用**
  - 作为“数据面”优先选择；能用就尽量用

## 3. 推荐组合（分阶段）

### MVP（推荐）

**浏览器插件（执行与工作台） + 本地/服务端控制平面（策略/调度/审计）**

- 插件负责：搜索/浏览、候选提取、低风险动作、评论草稿与人工确认
- 控制平面负责：主题配置、任务计划、配额限流、队列、报表、告警

### 规模化（第二阶段）

**控制平面（服务端） + 执行器集群（Playwright Worker） + 运营控制台**

- 每个账号绑定一个执行器（隔离环境与凭证）
- 所有动作通过统一队列与审计流水
- 更完善的风控：配额、速率、熔断、人工复核抽检

## 4. 为什么不建议“纯后台脚本直接互动”

缺少官方 API 时，纯后台模拟交互往往需要更强的自动化与更高的频率控制，**更容易触发平台风控且难以解释与审计**。把互动动作放到“真实浏览器 + 人工确认”可以显著降低风险，并提升可控性。
